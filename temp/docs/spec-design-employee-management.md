# Introduction

Specification for implementing employee creation and data editing in the Prodpol admin panel. The backend CRUD endpoints are already implemented; this spec covers frontend wiring, form validation, mutation setup, and error handling.

## 1. Purpose & Scope

Wire up the existing admin pages (`add.vue`, `edit.vue`) with the backend API to enable:

- Adding a new employee with name, email, phone, role, optional password, and photo
- Editing an existing employee's profile data (name, email, phone, role, photo)
- Proper validation feedback and error display on both forms

**Audience**: Frontend developer implementing the Vue 3 admin UI.
**Assumptions**: Backend endpoints at `/api/data/employees/` are stable and deployed.

## 2. Definitions

| Term | Definition |
|------|------------|
| JSON Patch | RFC 6902 format for partial updates: `[{op, path, value}]` |
| Snowflake ID | 64-bit time-sorted ID generated by backend via `ISnowflakeGenerator` |
| EmployeeCreate | Backend DTO for creating employees (required: Email, NameFirst, NameLast, PhoneNumber; optional: Password) |
| EmployeeRead | Backend response DTO with computed fields (FullName, RoleName, NormalizedEmail, etc.) |
| Pinia Colada | Vue query/mutation library (`@pinia/colada`) used for data fetching and mutations |
| Zod | Runtime schema validation library used in frontend types |
| EmployeeRoleRead | Role data including `id`, `roleName`, `displayName`, `employeesCount` |

## 3. Requirements, Constraints & Guidelines

### Requirements

- **REQ-001**: Add employee form must send `POST /api/data/employees` with `EmployeeCreate` JSON body on submit
- **REQ-002**: Edit employee form must load existing data via `GET /api/data/employees/{id}` and pre-populate fields
- **REQ-003**: Edit employee form must send `PATCH /api/data/employees/{id}` with JSON Patch body containing only changed fields
- **REQ-004**: Role selection must map the selected `roleName` (string) to a `roleId` (int?) by looking up the roles list
- **REQ-005**: Photo upload on add must associate the photo after employee creation via `PATCH /api/data/employees/photos/{id}`
- **REQ-006**: Photo upload on edit must call `PATCH /api/data/employees/photos/{id}` with the new image payload
- **REQ-007**: Validation errors from the backend (400 with `ValidationProblemDetails`) must be displayed inline on the form fields
- **REQ-008**: Successful create must navigate to the new employee's detail page (`/admin/employees/{id}`)
- **REQ-009**: Successful update must invalidate the employee query cache and stay on the edit page with a success indicator
- **REQ-010**: Duplicate email errors (returned as 400 validation error) must be shown on the email field

### Constraints

- **CON-001**: `Employee` model `RoleId` is `int option` ã cannot be assigned a string; must resolve `roleName` ã `roleId` client-side
- **CON-002**: Photo upload payload format must match what the backend expects (binary image data in the PATCH body)
- **CON-003**: Password field is create-only; no password editing on the edit form
- **CON-004**: All regex patterns on the frontend must match backend validation (email, name, phone patterns from `Employee.fs` and `EmployeeCreate.fs`)

### Patterns

- **PAT-001**: Use Pinia Colada `defineMutationOptions` for all create/update operations (see `employees.ts` and `employeeRoles.ts`)
- **PAT-002**: Use Zod schemas to parse API responses and validate form input client-side
- **PAT-003**: Error display should surface `ModelState` errors from `HttpValidationProblemDetails` responses

### PATCH `/api/data/employees/{id}` ã Update Employee

**Request body** (JSON Patch):
```json
[
  { "op": "replace", "path": "/nameFirst", "value": "NoweImie" },
  { "op": "replace", "path": "/phoneNumber", "value": "+48987654321" }
]
```

**Response 200** ã Updated `EmployeeRead` object.

**Error 400** ã `ValidationProblemDetails` with per-field errors.

### PATCH `/api/data/employees/photos/{id}` ã Update Photo

**Request**: Binary image data in body (content-type determined by image format).

**Response 200**: Success string.

### Frontend: `useEmployeeUpdate` Mutation (to be created)

```ts
export const useEmployeeUpdate = defineMutationOptions((emp: EmployeeRead) => ({
  key: ["employee", "update", emp.id],
  mutation: async (vars: {
    id: number;
    patch: Array<{ op: "replace"; path: string; value: unknown }>;
  }) => {
    const response = await fetch(`/api/data/employees/${vars.id}`, {
      method: "PATCH",
      headers: { "Content-Type": "text/json" },
      body: JSON.stringify(vars.patch),
    });
    if (!response.ok) throw new Error(await response.text());
    return employeeReadSchema.parse(await response.json());
  },
}));
```

## 5. Acceptance Criteria

- **AC-001**: Given a valid employee form on the add page, When the user clicks "Dodaj", Then a POST request is sent and on success the user is navigated to `/admin/employees/{id}`
- **AC-002**: Given the edit page for an existing employee, When the page loads, Then the form fields are pre-populated with the employee's current data
- **AC-003**: Given changed fields on the edit form, When the user clicks "Zapisz zmiany", Then a PATCH request with JSON Patch operations for only the changed fields is sent
- **AC-004**: Given a duplicate email on create or edit, When the form is submitted, Then the email field shows "Email is already present" error
- **AC-005**: Given a selected role in the EmployeeGroupSelector, When the form is submitted, Then the resolved `roleId` (int) is sent in the JSON Patch or EmployeeCreate payload
- **AC-006**: Given a selected photo file, When the employee is created/updated, Then the photo is uploaded via `PATCH /api/data/employees/photos/{id}`

## 7. Rationale & Context

- JSON Patch was chosen over PUT for updates because it allows partial updates without requiring the full employee object, and the backend already implements this pattern. The `EmployeesController.Update` method loads the existing entity, applies the patch, validates, and saves ã preventing unintended field overwrites.
- No backend changes are required because the existing `EmployeesController` and `PgEmployeeRepository` already handle all CRUD operations including duplicate email validation (within `UpdateAsync`).
- The `EmployeeGroupSelector` emits `roleName` (string) rather than `roleId` ã this must be resolved client-side by querying the roles list and finding the matching entry. This is because the selector displays names, not IDs.
- Photo upload is a separate PATCH endpoint rather than being included in the employee create body because photos involve binary data which doesn't fit the JSON payload format.

## 9. Examples & Edge Cases

### JSON Patch ã Update only changed fields

```ts
// Given form has changes to nameFirst and roleId
const patch = [];
if (form.firstName !== emp.nameFirst) {
  patch.push({ op: "replace", path: "/nameFirst", value: form.firstName });
}
if (form.groupId !== emp.roleId) {
  patch.push({ op: "replace", path: "/roleId", value: form.groupId });
}
// Send PATCH with: [{op:"replace", path:"/nameFirst", value:"Nowy"}, {op:"replace", path:"/roleId", value:3}]
```

### Edge case: Remove role assignment

```ts
// When user selects "Brak grupy" (no role):
patch.push({ op: "replace", path: "/roleId", value: null });
```

### Edge case: Duplicate email on update

The user changes email to one already taken by another employee ã backend returns 400:

```json
{
  "errors": {
    "Email": ["Email is already present"]
  }
}
```

This should be parsed and displayed as an inline error on the email field.

### Edge case: Photo too large

Frontend should enforce 2MB limit before upload attempt (already labeled in UI but not enforced).

### Edge case: Form loaded with `useRouteQuery` params on add page

The add page supports query params to pre-fill fields: `?email=...&name=...&phone=...&role=...`. This should be preserved when wiring up the form.
